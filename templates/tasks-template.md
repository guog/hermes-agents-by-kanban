# 任务：[功能名称]

**TASKS Key**：`[与 SPEC/PLAN 相同的 lowercase-kebab-case]`

**输入**：`[spec-<key>.md 固定链接]`、`[plan-<key>.md 固定链接]`

**前置条件**：对应 SPEC 与 PLAN 已通过当前 digest 门禁

**组织方式**：按阶段和用户故事分组，使每个故事可独立实现、独立测试

**稳定性**：任务 ID 在当前 PRD 的完整 `tasks/*.md` 集合内全局唯一；不同文件不得从 `T001` 重新编号。返工时保留既有 ID，只追加新 ID 或修改受影响任务，不重排无关 ID

## 任务格式（强制）

每个任务的首行必须严格使用：

```text
- [ ] T001 [P?] [US?] 使用动词描述可完成的工作，并包含精确文件路径
```

- `[P]`：仅当任务修改不同文件且不依赖未完成任务时使用。
- `[US1]`：仅用户故事阶段必须填写；准备、基础和收尾阶段不写故事标签。
- `T001` 只是格式示例；实际 ID 必须按完整 TASKS 集统一分配，并与其他 `task-<key>.md` 无冲突。
- 每个任务必须给出需求来源、`depends_on`、客观验收和测试。
- 没有新增自动化测试时，仍须写出可重复的验证命令/步骤和不新增测试的理由。

任务详细结构：

```markdown
- [ ] T001 [P] [US1] 在 `src/example.py` 实现 [具体行为]
  - 来源：US1、FR-001
  - depends_on：[]
  - 验收：[可观察、可判定的完成条件]
  - 测试：`[精确测试命令或验证步骤]`
```

## 依赖与执行顺序（必填）

### 阶段依赖

- 准备阶段：无依赖。
- 基础阶段：依赖准备阶段，并阻塞全部用户故事。
- 用户故事阶段：依赖基础阶段；故事间依赖必须明确，默认不隐含依赖。
- 收尾阶段：依赖计划交付的全部用户故事。

### 执行波次

| 波次 | 可执行任务 | 前置条件 | 并行理由 |
| --- | --- | --- | --- |
| 1 | T001、T002 | 无 | 不同文件、无未完成依赖 |
| 2 | T003 | T001 | 共享基础能力 |
| 3 | T004、T005、T007 | T003 | 不同文件、故事可独立推进 |

### 任务 DAG

```text
T001 ─→ T003 ─→ T004 ─→ T006
  └────────────→ T005 ─┘
T002
T003 ─→ T007 ─→ T008
```

| --- | --- | --- | --- |
| US1 / FR-001 | T005、T006 | T004 | 是/否 |
| US2 / FR-003 | T008 | T007 | 是/否 |

## 实施策略

1. 先完成准备与阻塞基础能力。
2. 优先完成 P1 用户故事，并独立验证 MVP。
3. 按优先级递增交付其余故事；每个故事完成时运行独立验证和相关回归。
4. 最后处理跨故事事项并运行 PLAN 中定义的完整门禁。
